# v2.4.x comparison notes for v2.4.5

## What changed across versions

| Area | v2.4 / v2.4.1 | v2.4.3-r2/r3 | v2.4.4 conservative | v2.4.5 |
|---|---|---|---|---|
| VM106 storage | default `nvme-vm` | optional `nvme-media` override | hard-fails if `nvme-media` missing | standard `nvme-vm-two` |
| VM107 storage | default `nvme-vm` | optional `nvme-media` override | hard-fails if `nvme-media` missing | standard `nvme-vm-two` |
| MLD M500 handling | not used for VM106/107 | tries to create `nvme-media` | tries to create `nvme-media`, but only if blank MLD detected | creates `nvme-vm-two`, fails early if unsafe |
| Pipeline failure handling | `run_required` exit-code bug present but less visible | same bug | same bug caused continued install after VM106/107 failure | fixed |
| Uptime Kuma | created monitors | created monitors | created active monitors for absent VM106 | disables optional monitors when absent |

## Why the five services are down

The services are not five separate service bugs. They share one missing host:

`192.168.50.106` / VM106 `docker-media`

v2.4.4 never created VM106 because `nvme-media` did not exist. The later service installers and Uptime Kuma only made the symptom visible.

## Fix strategy

Do not patch each service independently first. First make VM106 and VM107 reliably born:

1. Prepare `nvme-vm-two`.
2. Create VM106 on `nvme-vm-two`.
3. Create VM107 on `nvme-vm-two`.
4. Only then run Docker host prep, service install, service config, and Uptime Kuma monitors.

That is the v2.4.5 foundation.
